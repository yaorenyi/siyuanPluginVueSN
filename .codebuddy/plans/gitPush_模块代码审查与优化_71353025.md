## 用户需求

对 `src/features/gitPush` 模块的全部 TypeScript/Vue 代码进行深度审查，涵盖三个维度：

1. **逻辑漏洞与边界异常排查**：识别潜在 bug、内存泄漏、异常未处理、跨设备路径不一致等问题
2. **冗余逻辑与重复代码合并**：消除重复导入、重复函数定义、重复模式代码，提升可读性
3. **性能优化**：清理死代码、优化算法复杂度、修复资源泄漏

**核心约束**：所有修改不得改变现有业务逻辑与核心功能行为，不修改任何公开函数签名，输出代码须完全符合 CLAUDE.md 编码规范（if 花括号、文件头注释、统一入口原则、代码分层规则）。

## 审查结果概览

共发现 15 项问题，分为 5 类：

- **逻辑漏洞** 5 项（内存泄漏、跨设备路径 bug、异常未返回、进度状态卡死、加载失败不可重试）
- **冗余代码** 4 项（重复导入、重复函数、重复模式、调试日志残留）
- **死代码** 2 项（未使用的防抖 Map、未清理的缓存）
- **性能问题** 3 项（O(N*M) 查找、双重遍历、信号监听器泄漏）
- **健壮性** 1 项（BFS 符号链接无限循环）

## 技术栈

- Vue 3 + TypeScript + Vite（思源笔记插件）
- 纯函数级修复，不引入新依赖，不改变架构模式
- 修改范围限定在 `src/features/gitPush/` 目录内 7 个文件

## 实现方案

### 策略概述

按文件依赖关系分 4 批修改，每批聚焦一类问题，最小化跨文件上下文切换。所有修复均为等价变换或补全缺失逻辑，不改变函数签名和事件流。

### 详细修复清单

#### 批次 1：GitPushManager.ts + utils.ts — 逻辑漏洞修复 + 共享工具提取

| 编号 | 问题 | 修复方案 |
| --- | --- | --- |
| A1 | `destroy()` 未清理 `networkWaitQueue`（内存泄漏） | 第1385行后添加 `this.networkWaitQueue.length = 0` |
| A2 | `checkCanPushToCloud` 使用 `project.path` 而非 `resolveValidPath`（跨设备 bug） | 第1044行 `project.path` → `resolveValidPath(project)` |
| D3 | `tryRemoteOp` 重试延时中 signal 监听器泄漏 | 第430行定时器 resolve 后添加 `signal.removeEventListener("abort", onAbort)` |
| E1 | `scanForGitRepos` BFS 不处理符号链接（潜在无限循环） | 第1015行条件增加 `&& !entry.isSymbolicLink()` |
| B3 | 平台远程名称提取模式重复（GitPushManager 内 3 处） | utils.ts 新增 `getProjectRemoteNames(project)` 函数，`remoteOpAll`/`fetchAllForProject`/`checkPushStatus` 3 处调用替换 |


**`getProjectRemoteNames` 签名**（utils.ts 新增）：

```typescript
export function getProjectRemoteNames(project: GitProject): { key: PlatformKey, name: string }[]
```

遍历 `PLATFORM_META`，返回所有已配置远程的 `{ key, name }` 数组。需要从 `./types` 额外导入 `PlatformKey` 类型。

#### 批次 2：useGitOps.ts + useCommitLog.ts — 异常处理逻辑补全

| 编号 | 问题 | 修复方案 |
| --- | --- | --- |
| A3 | `remoteOpAll` catch 块未 return（返回 undefined） | catch 块末尾构造并 return 错误结果对象 `{ [key]: { ok: false, stdout: "", stderr: msg } }` |
| A4 | `remoteOpSingle` 缺少 try/catch（失败时进度卡在 "pushing"） | 添加 catch 块：设置进度为 "fail"、构造错误输出条目、pruneRecordCache、不 re-throw（与 remoteOpAll 一致） |
| A5 | `handleExpand` 失败后无法重试 | catch 中 `expandedProjects.value.delete(projectId)` + `console.warn`，不 re-throw |


**A3 返回值构造**：

```typescript
// catch 块末尾
const failResult: Record<string, any> = { success: false }
for (const key of Object.keys(initProg)) {
  failResult[key] = { ok: false, stdout: "", stderr: String((e as any)?.message || e || "未知错误") }
}
return failResult
```

**A4 catch 块结构**：

```typescript
catch (e: any) {
  progressRef.value = {
    ...progressRef.value,
    [id]: { ...progressRef.value[id], [target]: "fail" },
  }
  const pm = PLATFORM_META.find((m) => m.key === target)
  outputsRef.value[id] = [{
    platform: target,
    label: pm?.label ?? target,
    ok: false, skipped: false,
    duration: Date.now() - startedAt,
    summary: e?.message?.split("\n")?.[0]?.trim() || "失败",
    fullStdout: "",
    fullStderr: e?.message || String(e),
  }]
  pruneRecordCache(outputsRef.value)
}
```

## 实现备注

- **性能热点**：`useGitStats.projectStats` computed 已是单次遍历优化设计，本次不改动其结构（改动风险高、收益低）
- **响应式安全**：所有 Record ref 的修改沿用现有的浅拷贝模式（`{ ...ref.value, [key]: val }`），不引入新的响应式模式
- **错误处理一致性**：`remoteOpAll` 和 `remoteOpSingle` 修复后均采用 swallow + log 模式（不 re-throw），与现有事件处理器调用方式一致（Vue 事件绑定无 catch）
- **日志规范**：移除 console.log 调试日志，保留 console.error/warn（错误诊断必要）
- **向后兼容**：所有函数签名不变，事件名不变，props 不变，模板绑定不变
