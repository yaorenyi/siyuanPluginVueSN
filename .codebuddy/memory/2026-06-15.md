## gitPush 刷新性能优化
- **文件**: `src/features/gitPush/index.vue` (`handleRefresh`)
- **问题**: 单项目刷新 6 步完全串行（refreshRemotes → loadPushStatus → loadWorkingTree → loadCommitLog → loadBranches → loadStashList），大仓库或内网环境下 `git update-index --refresh` 等命令慢时会阻塞整个刷新
- **修改**: 改为 2 组并行执行，与 `silentRefreshAll` 保持一致模式 — 第 1 组 `Promise.all([refreshRemotes, loadPushStatus])`，第 2 组 `Promise.all([loadWorkingTree, loadCommitLog, loadBranches, loadStashList])`
- **关键发现**: 刷新流程所有 git 命令都是本地操作（`git status`、`git log`、`git branch` 等），不依赖网络。慢的原因是大仓库 `git update-index --refresh` 的 I/O 耗时 + 串行阻塞模式

## gitPush 批量刷新批次化（修复 silentRefreshAll 拥堵）
- **文件**: `src/features/gitPush/index.vue` + `src/features/gitPush/types/index.ts`
- **问题**: `silentRefreshAll`/`onMounted`/watchers 中 `Promise.all(projList.map(...))` 让所有项目同时涌入 3 槽位 git 信号量，10 个项目 × 8 命令 = 80 个命令排队。`checkPushStatus` 内部对 remote 的检查也是串行 for 循环
- **修改 1** (index.vue): 添加 `batchProcess` 工具函数，将 `silentRefreshAll`、`onMounted` 定时器、`watch(activeCategory)`、`watch(currentView)` 四处批量刷新点全部改为 `batchProcess(items, 3, fn)` 批次处理（每批 3 个项目匹配信号量上限）
- **修改 2** (types/index.ts): `checkPushStatus` 中 3 个 remote 的检查从串行 `for...await` 改为 `Promise.all(remoteChecks.map(...))` 并行执行，减少单项目内串行等待
