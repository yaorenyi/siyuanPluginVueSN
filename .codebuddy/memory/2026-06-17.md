## GitPush AI 提交信息生成乱输出修复（v3 — 最终修复）

### 根因（DeepSeek API 文档审查）
DeepSeek API 规定：**必须显式传递 `thinking: {"type": "disabled"}` 才能关闭思考模式**。若省略该参数，支持思考的模型（v4 系列/reasoner）**默认开启思考**（ref: github.com/NousResearch/hermes-agent/issues/15700 + api-docs.deepseek.com）。

v2 代码传入 `enableThinking: false` 时，`buildRequestBody` 直接省略了 `thinking` 字段 → DeepSeek 仍默认开启思考 → 输出推理文本 → 正则提取失败 → 降级启发式。

### 修改（2 个文件）
- **`utils/aiApi.ts`** `buildRequestBody()`：新增显式禁用分支 —— 当 `enableThinking === false` 且是 DeepSeek 模型时，发送 `thinking: {"type": "disabled"}`（而非省略）。中间变量 `deepseekThinking` 拆分为 `isDeepSeek` + 独立判断
- **`features/gitPush/types/index.ts`**：恢复 `enableThinking: false`（现在已正确生效，会在请求体中显式禁用思考）

### 数据流
```
enableThinking: false
  → buildRequestBody 判断 isDeepSeek && enableThinking === false
  → 请求体: { thinking: { type: "disabled" } }
  → DeepSeek API 不输出 reasoning_content
  → 正则匹配 type: 描述 成功
```

- 构建通过，零 lint 错误

## GitPush 统计视图方案二：消除重复 rev-parse（loadPushStatus/loadWorkingTree 共用分支名）

### 问题
上轮优化已将 stats watch 精简为 2 个 git 命令/项目（pushStatus + workingTree），但 `loadPushStatus` 和 `loadWorkingTree` 各执行一次 `git rev-parse --abbrev-ref HEAD` 获取分支名，对同一项目重复执行相同命令。50 项目 ≈ 50 次冗余 rev-parse。

### 修改（3 个文件）
- **`types/index.ts`**：
  - 新增 `getBranch(projectPath)` 公开方法（轻量 rev-parse 封装）
  - `checkPushStatus(id, opts?)`：`opts.branch` 传入时跳过内部 rev-parse
  - `getWorkingTreeStatus(projectPath, opts?)`：`opts.branch` 传入时跳过内部 rev-parse
- **`composables/useGitPush.ts`**：
  - `loadPushStatus(id, opts?)` / `loadWorkingTree(id, skipRefresh, branch?)` 新增可选 branch 参数透传
  - 新增 `loadStatsData(id)` 复合函数：`getBranch()` 一次 → 分发给两个加载函数
  - 导出 `loadStatsData` 供 index.vue 使用
- **`index.vue`**：stats watch 改为调用 `loadStatsData(p.id)` 替代两个独立 load 调用

### 收益
- git 命令数/未缓存项目：2 → 1.5（rev-parse 从 2 次降为 1 次，status + rev-list 不变）
- 50 项目：节省 ~50 次 git rev-parse 调用

### 构建验证
- `npx vite build` 通过，零新增 lint 错误
- 仅 1 个预存 HINT（noPlatformProjects 已弃用）
